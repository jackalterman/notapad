## C1 — Chunked large-file loading is corruptible by a second open · CRITICAL

**Status:** `[ ]` OPEN
**File:** `notapad.py` · `_load_large_file_chunked()` and `open_file()`
**Severity:** Critical — data corruption; content from two different files can be interleaved in the editor

### Description
`_load_large_file_chunked` queues potentially dozens of `after_idle` callbacks that each check
`if not self._loading` before inserting the next chunk. If the user opens a second file before
the first finishes loading, `open_file()` clears the text widget and calls
`_load_large_file_chunked` again — setting `self._loading = True` again. The OLD queued
callbacks from the first load now see `self._loading == True` and continue inserting chunks of
the **old file** into the partially-loaded content of the new file.

### Fix
Introduce a monotonically incrementing generation counter:
```python
self._load_generation = 0  # in __init__
```
In `open_file()` / `_load_large_file_chunked()`:
```python
self._load_generation += 1
gen = self._load_generation   # snapshot for this load

def _insert(i: int):
    if gen != self._load_generation:
        return  # stale callback — a new load has started; abort
    ...
```
Additionally, disable user input during loading:
```python
self.text.config(state="disabled")   # before first chunk
# ... in finalization:
self.text.config(state="normal")
```

## C2 — Find/Replace dialogs leave dead widget reference after Escape / Cancel · CRITICAL

**Status:** `[ ]` OPEN
**File:** `notapad_app/dialogs.py` · `open_find()` (~line 97) and `open_replace()` (~line 215)
**Severity:** Critical — TclError on next search after dialog is keyboard-dismissed

### Description
Both `open_find()` and `open_replace()` define a `_cleanup()` function that resets
`app.search_count_lbl` and `app.search_window` to `None`. However, `_cleanup()` is only
wired to the `WM_DELETE_WINDOW` protocol. Every other close path — Escape binding
(`top.bind("<Escape>", lambda e: top.destroy())`), and the Cancel button
(`_create_styled_button(footer, "Cancel", top.destroy, t)`) — calls `top.destroy()` directly,
**bypassing `_cleanup()`**.

After the dialog is dismissed via Escape or Cancel, `app.search_count_lbl` still points to
the destroyed `tk.Label`. The next call to `_update_match_label()` does
`self.search_count_lbl.config(...)` on the dead widget, raising a `TclError` or silently
failing depending on Tkinter version. The inline find bar (`open_find_bar`) is unaffected
because it owns its own label, but any code path that uses the dialog sets `search_count_lbl`
to the dialog's label, leaving the app in a broken state.

### Fix
Introduce a single close helper in each dialog that always calls `_cleanup()`:
```python
def _close():
    _cleanup()
    top.destroy()

# Wire ALL close paths to _close():
top.protocol("WM_DELETE_WINDOW", _close)
top.bind("<Escape>", lambda e: _close())
_create_styled_button(footer, "Cancel", _close, t)
```
No other changes needed.

## C3 — save_file() silently swallows errors — no user feedback on failure · CRITICAL

**Status:** `[ ]` OPEN
**File:** `notapad.py` · `save_file()` (~line 573)
**Severity:** Critical — data loss; user believes file was saved when it was not

### Description
`save_file()` catches all exceptions and silently returns `False`:
```python
except Exception: return False
```
`is_modified` is only cleared on the success path, so the dirty marker (`•`) technically
remains correct — but the user is given **no feedback** that the save failed. If they then
close the app, `_confirm_discard()` sees `is_modified == True` and prompts them to save again,
which will fail again silently if the underlying problem (permissions, disk full, path gone)
persists. In the worst case, a user who dismisses the "save changes?" prompt loses their work.

### Fix
Add an error dialog to the except block, matching the pattern already used in `open_file()`:
```python
except Exception as exc:
    messagebox.showerror(APP_NAME, f"Could not save file:\n{exc}")
    return False
```

## H1 — choose_font() apply() crashes on non-integer font size · HIGH

**Status:** `[ ]` OPEN
**File:** `notapad_app/dialogs.py` · `choose_font()` → `apply()` (~line 341)
**Severity:** High — unhandled crash in font dialog

### Description
The `apply()` callback calls `int(size_var.get())` without error handling. If the user
types letters or clears the Spinbox field and clicks OK, a `ValueError` propagates uncaught.
The live preview `refresh()` wraps the same call in `try/except`, showing the inconsistency.

### Fix
```python
def apply():
    try:
        size = int(size_var.get())
    except ValueError:
        messagebox.showwarning(APP_NAME, "Please enter a valid font size.", parent=top)
        return
    app._font_family = fam_var.get()
    app._font_size   = size
    app._apply_font()
    top.destroy()
```

## H2 — Text widget not disabled during chunked large-file loading · HIGH

**Status:** `[ ]` OPEN
**File:** `notapad.py` · `_load_large_file_chunked()` (~line 533)
**Severity:** High — document corruption; user can type/paste into a partially-loaded file

### Description
Between `after_idle` chunk insertions the event loop is live. The user can type, paste, or
delete content into the text widget while the load is still in progress. Since inserts happen
at `tk.END` and user edits land at `tk.INSERT`, the two streams interleave, corrupting the
document mid-load with no warning.

### Fix
Set `self.text.config(state="disabled")` before starting the first chunk, and restore
`state="normal"` in the finalization block (after `self._loading = False`). Also update the
status bar label to indicate loading is in progress. See also C1 for the generation-counter
fix that should be applied at the same time.

## H3 — _update_status() word-counts the entire document on every keystroke · HIGH

**Status:** `[ ]` OPEN
**File:** `notapad.py` · `_update_status()` (~line 594)
**Severity:** High — UI becomes sluggish/unusable on files > ~200 KB

### Description
`_update_status()` is called from `_on_key()` (every `<KeyRelease>`) and
`<ButtonRelease>`. Inside, it does:
```python
content = self.text.get("1.0", tk.END)
self.status_words.config(text=f"Words: {len(content.split())}  Chars: {len(content)-1}")
```
`text.get("1.0", tk.END)` copies the **entire document** out of the Tcl layer into Python,
then `split()` walks every character. On a 1 MB file this can easily take 50–100ms,
introducing perceptible input lag on every single keypress.

### Fix
Separate the fast path (cursor position) from the slow path (word/char count). The cursor
position must update immediately; word count does not need to be real-time:
```python
def _update_status(self, event=None):
    try:
        row, col = self.text.index(tk.INSERT).split(".")
        self.status_pos.config(text=f"Ln {row}, Col {int(col)+1}")
    except Exception:
        pass
    self._schedule_bracket_match()
    self._schedule_word_highlight()
    self._schedule_word_count()   # new debounced helper

def _schedule_word_count(self):
    if hasattr(self, "_wc_after") and self._wc_after:
        self.root.after_cancel(self._wc_after)
    self._wc_after = self.root.after(300, self._do_word_count)

def _do_word_count(self):
    self._wc_after = None
    try:
        content = self.text.get("1.0", tk.END)
        self.status_words.config(text=f"Words: {len(content.split())}  Chars: {len(content)-1}")
    except Exception:
        pass
```

## H4 — Bracket matching walks the document character-by-character via Tcl · HIGH

**Status:** `[ ]` OPEN
**File:** `notapad.py` · `_do_bracket_match()` (~line 910)
**Severity:** High — UI freeze on files with deeply nested or unmatched brackets

### Description
`_do_bracket_match()` searches for the matching bracket by iterating one character at a time:
```python
while self.text.compare(idx, "<", tk.END):
    c = self.text.get(idx)           # one Tcl round-trip per char
    ...
    idx = self.text.index(f"{idx}+1c")  # another Tcl round-trip
```
An unmatched `(` at the top of the file causes this loop to traverse every character until
`tk.END`, each iteration requiring two Tcl bridge calls. On a 500-line file this is
approximately 20,000 individual Tcl calls in the idle handler — enough to noticeably freeze
the UI. The 80ms debounce helps between keystrokes but not during the traversal itself.

### Fix
Fetch a bounded text chunk into Python once and iterate over the Python string:
```python
LOOK_LIMIT = 5000  # characters max look-ahead/behind

if ch in OPEN:
    chunk = self.text.get(f"{pos}+1c", f"{pos}+{LOOK_LIMIT}c")
    depth = 1
    for i, c in enumerate(chunk):
        if c == ch: depth += 1
        elif c == target:
            depth -= 1
            if depth == 0:
                # Convert offset back to Tk index for tag_add
                match_idx = self.text.index(f"{pos}+{i+1}c")
                self.text.tag_add("bracket_match", pos, f"{pos}+1c")
                self.text.tag_add("bracket_match", match_idx, f"{match_idx}+1c")
                return
```
This reduces Tcl round-trips from O(n chars) to O(1) for the fetch plus 2 for the tags.

## H5 — Status bar separator hardcoded white in dark mode · HIGH

**Status:** `[ ]` OPEN
**File:** `notapad.py` · `_build_statusbar()` (~line 253)
**Severity:** High — visually jarring bright line in dark mode

### Description
The top separator of the status bar is created with a hardcoded color:
```python
tk.Frame(self.statusbar, height=1, bg="#e0e0e0").pack(side=tk.TOP, fill=tk.X)
```
`apply_theme()` themes all `isinstance(child, tk.Frame)` children of the statusbar, but
only when they have `winfo_width() == 1` (separator heuristic). This 1px-tall Frame has
`winfo_height() == 1`, not `winfo_width()`. The check therefore misses it, leaving a
bright `#e0e0e0` line between the editor and the status bar in dark mode.

### Fix
Store a reference and update it explicitly in `apply_theme()`:
```python
# In _build_statusbar():
self.status_top_sep = tk.Frame(self.statusbar, height=1)
self.status_top_sep.pack(side=tk.TOP, fill=tk.X)

# In apply_theme():
self.status_top_sep.configure(bg=t["sep"])
```

## H6 — _confirm_discard() gives no feedback when save fails during a discard flow · HIGH

**Status:** `[ ]` OPEN
**File:** `notapad.py` · `_confirm_discard()` (~line 472)
**Severity:** High — silent data loss; user clicks "Yes" to save but nothing happens

### Description
```python
def _confirm_discard(self) -> bool:
    ...
    result = messagebox.askyesnocancel(...)
    if result is True: return self.save_file()  # returns False on failure — silently!
    ...
```
When the user chooses "Yes" (save), `save_file()` is called. If it fails (see C3), it
returns `False`. `_confirm_discard()` propagates that `False` back to the caller
(`new_file`, `open_file`, `close_file`, `on_close`), which all treat `False` as "user
cancelled." The pending operation is silently abandoned with no explanation.

### Fix
Check the return value explicitly and inform the user:
```python
if result is True:
    if not self.save_file():
        messagebox.showerror(APP_NAME,
            "Save failed. The operation has been cancelled.\n"
            "Please check file permissions and try again.")
        return False
    return True
```
This fix is only fully effective once C3 is also fixed (so `save_file()` actually reports
the error to the user independently as well).

## H7 — Ctrl+H (Replace) and Ctrl+G (Go To Line) not bound on root · HIGH

**Status:** `[ ]` OPEN
**File:** `notapad.py` · `_bind_events()` (~line 332–338)
**Severity:** High — keyboard shortcuts silently fail when focus is not in the text widget

### Description
Several Edit menu shortcuts are only bound on the text widget (`t.bind(...)`) and not on
the root window (`r.bind(...)`):
```python
t.bind("<Control-h>", lambda e: (dialogs.open_replace(self), "break")[1])
t.bind("<Control-g>", lambda e: (dialogs.goto_line(self), "break")[1])
```
When focus is on a menu button, a dialog title bar, or elsewhere outside the text widget
(e.g. after dismissing a dialog that doesn't restore focus), these shortcuts produce no
response. Other shortcuts like Ctrl+F, Ctrl+N, Ctrl+S, and F3 are correctly bound on
both `t` and `r`.

### Fix
Add root-level bindings to match:
```python
r.bind("<Control-h>", lambda e: dialogs.open_replace(self))
r.bind("<Control-g>", lambda e: dialogs.goto_line(self))
```

## B4 — Horizontal scrollbar invisible when word-wrap is disabled · HIGH

**Status:** `[x]` FIXED 2026-04-13
**File:** `notapad.py` · `_build_editor()` and `toggle_wrap()`
**Severity:** High — no way to scroll long lines when wrap is off

### Description
`scrollbar_x` is created as a child of `self.root` but never initially packed.
`toggle_wrap()` tries to show it with:
```python
self.scrollbar_x.pack(side=tk.BOTTOM, fill=tk.X, before=self.editor_frame)
```
`self.editor_frame` is **not** a direct child of `root` — it lives inside `self.pane`
(a `ttk.Panedwindow`). Pack's `before=` argument requires a sibling widget in the same
container. Passing a non-sibling either silently fails or raises a Tcl error, so the
scrollbar is never made visible.

### Fix
Change `before=self.editor_frame` → `before=self.pane` in `toggle_wrap()`, since
`self.pane` **is** a direct child of root and is the correct anchor point:
```python
self.scrollbar_x.pack(side=tk.BOTTOM, fill=tk.X, before=self.pane)
```

**Status:** `[x]` FIXED 2026-04-13

## B1 — `re` module shadowed in `open_replace()` · CRITICAL

**Status:** `[x]` FIXED 2026-04-12
**File:** `notapad_app/dialogs.py` · `open_replace()`
**Severity:** Critical — Replace All is completely broken

### Description
`import re` at the module top is silently overwritten inside `open_replace()`:
```python
re = tk.Entry(body, textvariable=rv, ...)   # shadows the re module
```
Every subsequent call to `re.findall()` and `re.sub()` inside `do_replace_all()` hits the
`tk.Entry` widget instead of the regex engine. The result is either a silent `AttributeError`
or a nonsensical result depending on Python version / Tkinter internals.

### Fix
Rename the Entry variable. Convention already used elsewhere in the file:
```python
# Before
re = tk.Entry(body, textvariable=rv, width=32, **ent_opts)
re.grid(row=3, ...)

# After
rep_ent = tk.Entry(body, textvariable=rv, width=32, **ent_opts)
rep_ent.grid(row=3, ...)
```
No other changes required — `rv` (the `StringVar`) is what the rest of the function reads.

## B2 — Replace All destroys undo history · HIGH

**Status:** `[x]` FIXED 2026-04-12
**File:** `notapad_app/dialogs.py` · `do_replace_all()`
**Severity:** High — data-loss class bug (undo broken after any Replace All)

### Description
`do_replace_all()` reconstructs the full buffer and replaces it wholesale:
```python
app.text.delete("1.0", tk.END)
app.text.insert("1.0", new_content)
```
This nukes the entire `tk.Text` undo stack. After a Replace All the user cannot undo
back past that point — all prior edits are gone from history.

### Fix
Perform replacements individually in reverse-index order so each is a discrete undo unit,
or bracket the operation with `edit_separator` and a single atomic replacement:
```python
app.text.config(autoseparators=False)
app.text.edit_separator()
app.text.delete("1.0", tk.END)
app.text.insert("1.0", new_content)
app.text.edit_separator()
app.text.config(autoseparators=True)
```
The cleaner approach is to collect all match spans in reverse order and replace each
individually — this produces one undo step per replacement and keeps the stack intact.

## B3 — Syntax highlight runs synchronously on every keystroke · HIGH

**Status:** `[x]` FIXED 2026-04-12
**File:** `notapad.py` · `_schedule_highlight()` and `notapad_app/editor.py` · `apply_highlight()`
**Severity:** High — causes UI stutter on any non-trivial file

### Description
`_schedule_highlight()` calls `apply_highlight(self)` directly and synchronously:
```python
def _schedule_highlight(self):
    apply_highlight(self)      # no after(), no debounce
```
The `_hl_after` attribute is tracked and set to `None` inside `apply_highlight()`, but
`root.after()` is never called — the scheduling infrastructure exists but is unused.

### Fix
Implement the debounce that the attribute already anticipates:
```python
def _schedule_highlight(self):
    if self._hl_after:
        self.root.after_cancel(self._hl_after)
    self._hl_after = self.root.after(120, lambda: apply_highlight(self))
```
Inside `apply_highlight`, the existing `app._hl_after = None` line at the top is then
correct — it clears the pending ID once the callback fires.
120ms is the sweet spot: imperceptible to humans, eliminates per-keystroke firing.
